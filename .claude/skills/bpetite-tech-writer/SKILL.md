# bpetite Technical Writer

This skill governs all documentation work for the bpetite project. It covers three output modes: narrative markdown docs (D1–D4), Google-style docstring passes (D5–D8), and inline rationale comments.

Before writing a single word, identify which mode applies:

| Mode                   | When                                         | Triggered by                                                               |
| ---------------------- | -------------------------------------------- | -------------------------------------------------------------------------- |
| **Markdown narrative** | Writing D1–D4 standalone docs                | "Write D3", "draft core algorithm doc", "write the index"                  |
| **Docstring**          | Adding or editing module/function docstrings | "Docstring pass on \_trainer.py", "add worked example to module docstring" |
| **Inline comment**     | One-line rationale at a specific call site   | "Add rationale at sort_keys", "comment the mkstemp call"                   |

Apply the mode-specific section below. Rules under **All Modes** apply everywhere without exception.

---

## All Modes — Non-Negotiable Rules

### Voice

Cold, precise, technical, authoritative. Present indicative throughout.

- CORRECT: "The trainer selects the lexicographically smallest tied pair."
- WRONG: "The trainer will select..." / "The trainer can select..." / "The trainer should select..."

No filler. No hedging. No apologetic preambles. Start with the substance.

### Vocabulary lock

Use PRD terminology exactly. Introducing a synonym for a locked term is a bug, not a style choice.
Locked terms are introduced once in `docs/phase-2/index.md`. Use them identically everywhere else.

| Locked term              | Never substitute with                    |
| ------------------------ | ---------------------------------------- |
| `mergeable_vocab_size`   | "merge vocab size", "learned vocab size" |
| `vocab_size`             | "vocabulary size"                        |
| `merge rank`             | "merge order", "merge index"             |
| `pre-tokenizer`          | "tokenizer", "splitter"                  |
| `chunk`                  | "token", "piece", "segment"              |
| `base byte`              | "byte token", "byte character"           |
| `reserved special token` | "special token", "eos token"             |

### Protocol Zero — absolute prohibitions

These apply to every character of committed output. No exceptions.

- Zero emojis.
- Zero AI attribution. No "generated by Claude", "AI-assisted", co-authored-by trailers, or self-referential AI language in committed docs.
- No future tense ("will", "can") in place of present-tense assertions.
- No general BPE theory. bpetite documents bpetite's specific choices. Link out; do not survey alternatives.
- No duplication of PRD content. Cite FR numbers; do not re-derive requirements.

### Citation and reference discipline

Every normative claim cites the relevant FR:

> "The trainer never merges across chunk boundaries (FR-7)."

Source-file references use `file_path:line_number`:

> "See `src/bpetite/_trainer.py:42` for the merge application loop."

Cross-links use relative paths only. No absolute URLs for internal references.

### Scope enforcement

If a request touches out-of-scope content, stop and flag it. Do not silently absorb scope drift.

| In scope — Phase 2                                                    | Out of scope — flag and refuse                                    |
| --------------------------------------------------------------------- | ----------------------------------------------------------------- |
| `_constants.py`, `_pretokenizer.py`, `_trainer.py`, `_persistence.py` | `_encoder.py`, `_decoder.py`, `_tokenizer.py` (Phase 3)           |
| `tests/fixtures/`, `tests/conftest.py`                                | CLI behavior, CLI contract (Phase 4)                              |
| Artifact Schema v1                                                    | Benchmark scripts and results (Task 4-4)                          |
| Phase 2 test suite behavior                                           | Installation, quickstart, end-user CLI examples (Task 4-5 README) |

## Mode: Markdown Narrative (D1–D4)

### YAML frontmatter

Every narrative doc opens with this block. Fill all fields. Do not omit any.

```yaml
---
title: <Doc title>
description: <One-line description>
slug: <kebab-case slug>
order: <integer>
category: Phase 2
published: true
---
```

### Seven-section template

Every narrative doc follows this skeleton in this order. Do not reorder sections. Do not skip sections. Do not merge sections. Do not add sections without a documented reason.

```
# <Title> — <one-sentence purpose>

## TL;DR
<Exactly three bullets. One sentence each. No sub-bullets.>

## What lives here
<File-by-file table: | File | Purpose |>

## Key invariants
<FR-keyed table: | FR | Invariant | Consequence if violated |>

## Walkthrough
<Runnable worked example or traced execution. No pseudocode.>

## Failure modes
<Table: | Failure | Exception type | FR | Caught by |>

## Related reading
<Relative links to sibling docs, the PRD, and relevant source files.>
```

### Formatting rules

- Soft 100-character prose wrap.
- Code blocks hard-wrap at 88 characters.
- Tables over bullets for invariants, schema fields, failure modes, and fixture descriptions.
- ASCII diagrams only, inside code fences. One per doc maximum. No Mermaid, no SVG, no PlantUML.
- Every code example must be copy-pasteable Python that runs against the current repo state. No pseudocode.

### Audience lens by doc

| Doc               | Primary audience                          | Required depth                                                                                                                                 |
| ----------------- | ----------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------- |
| D1 index          | Portfolio reviewers                       | Narrative, 10-min read. Link-forward only; do not explain mechanics here.                                                                      |
| D2 core algorithm | Portfolio reviewers + future contributors | Worked example required. Trace one full merge step end-to-end from corpus bytes to updated vocab.                                              |
| D3 persistence    | All three audiences                       | Field-by-field schema table. Full loader validation checklist numbered to mirror loader code. Both determinism gates cited by exact test name. |
| D4 fixtures       | Future contributors                       | Per-file purpose table. Byte-count invariants. Whitespace-preservation gotcha documented.                                                      |

### Silent failure mode checklist

Before finalizing D2 or D3, verify each of these is named by exact test name and explained in the correct doc:

- [ ] **Wrong tie-break ordering** → `tests/test_trainer.py::test_train_tie_breaking_selects_lexicographically_smallest` → D2 Core Algorithm
- [ ] **Missing `sort_keys=True` in serializer** → `tests/test_persistence.py::test_same_state_saved_twice_produces_identical_bytes` → D3 Persistence
- [ ] **Cross-device temp-file atomic replace** → No automated test; document as a code-level constraint with rationale in D3 Persistence

### Sequencing rule

Write docs in this order. Do not start a doc before the one above it is in a stable draft.

1. **D3 Persistence** — Highest value, most mature content, locks the markdown template for all docs that follow.
2. **D2 Core Algorithm** — Highest authoring effort. Template from D3 must be known-good before D2 starts.
3. **D4 Fixtures** — Smallest doc. Depends on the glossary settled in D2.
4. **D1 Index** — Last. Written after D2–D4 are stable because it links all three.

Docstring passes (D5–D8) start only after D1–D4 are merged. Docstrings cross-link to markdown anchors; anchors must be final before docstrings reference them.

## Mode: Docstring (D5–D8)

Use Google-style docstrings throughout. The existing docstrings in `_trainer.py` and `_persistence.py` are production-grade. D5 and D6 are touch-up passes, not rewrites. Match the existing register.

### Per-file instructions

**D5 — `src/bpetite/_trainer.py`**
Add an inline worked-example block in the module docstring tracing one merge step end-to-end through pair counting, tie-break selection, and replacement. Verify the `ProgressEvent` docstring states the event schedule precisely: start, every 100 merges completed, completion.

**D6 — `src/bpetite/_persistence.py`**
Add a cross-link to `docs/phase-2/persistence.md` in the module docstring. Add a one-line rationale comment at the `sort_keys=True` call site and at the `tempfile.mkstemp(dir=...)` call site.

**D7 — `src/bpetite/_pretokenizer.py`** (P2 — defer if Phase 3 has started)
Add a two-line rationale for why `regex` is used instead of stdlib `re` in the module docstring. Add one worked example showing input string → chunk list.

**D8 — `src/bpetite/_constants.py`** (P2 — defer if Phase 3 has started)
Add one line per constant in the module docstring explaining which modules would otherwise drift if the constant were not centralized.

### Docstring constraints

- After any docstring edit, `uv run mypy --strict` and `uv run ruff check .` must remain green. Verify before committing.
- Docstring cross-links reference `docs/phase-2/<filename>.md#<anchor>` using the exact anchor slugs from the published docs. Do not write cross-links before D1–D4 anchors are finalized.

## Mode: Inline Comment

One-line rationale only. State the why, not the what. The code already states the what.

- CORRECT: `# Same-directory temp file prevents cross-device rename on network mounts.`
- WRONG: `# Write to a temporary file first.` (restates the code — no information)
- WRONG: `# This is important for correctness.` (no diagnostic value)

Inline comments in `_persistence.py` are the primary target for this mode.
Do not add inline comments to test files.

## Quick-Reference Anti-Pattern Table

| Anti-pattern                                  | Correct behavior                             |
| --------------------------------------------- | -------------------------------------------- |
| Future tense: "will select", "can merge"      | Present indicative: "selects", "merges"      |
| Synonym for a locked vocabulary term          | Use the locked term exactly                  |
| Section added outside the 7-section template  | Stop, flag it, get explicit approval         |
| Pseudocode in a code block                    | Runnable Python only, 88-char wrap           |
| Absolute URL for an internal cross-link       | Relative path only                           |
| Explaining general BPE theory                 | Link out; document bpetite's specific choice |
| Restating PRD content verbatim                | Cite the FR number; do not re-derive         |
| Emojis anywhere in committed output           | Prohibited. No exceptions.                   |
| AI attribution language                       | Prohibited. Protocol Zero.                   |
| Writing D2 or D1 before D3 is in stable draft | Follow the sequencing rule                   |
| Starting D5–D8 before D1–D4 are merged        | Docstring cross-links require stable anchors |
| Inline comment that restates the code         | State the why; the code states the what      |
